### 2. 主要卡点与破局

**卡点1: Task状态机设计复杂**

- **问题**: 10+2状态转移，Owner锁定逻辑容易死锁
- **破局**: 参考Claude Code架构，引入状态转移表 + 严格的前置条件校验
- **工具**: 状态机可视化 + 41个单元测试覆盖所有转移路径

**卡点2: 多Agent协作上下文污染**  

- **问题**: 父Agent context被子Agent transcript污染，影响主链路
- **破局**: 每个sub-agent独立context window，通过消息传递而非共享状态协作
- **工具**: transcript隔离 + Builder-Validator严格分离

**卡点3: 飞书API限流与容错**

- **问题**: 真实API调用有频率限制，开发阶段经常被限流
- **破局**: 三级降级策略：真实API → lark-cli → 本地fallback
- **工具**: 自研工具链 + graceful degradation

**卡点4: Flutter多端构建环境差异**

- **问题**: iOS/Android/macOS/Windows构建依赖和配置复杂
- **破局**: 编写flutter_check.sh统一构建验证脚本，CI自动检查
- **工具**: 自动化脚本 + 平台差异抽象

## 四、本周随手记

### 开发心得

- **AI协作**: Claude在代码生成上很强，但架构设计和业务逻辑梳理还是需要人来主导。最佳实践是AI写初版，人负责架构和边界逻辑。
- **测试策略**: E2E测试比单元测试更能发现真正的问题，特别是状态机和异步流程。
- **飞书生态**: 官方SDK很完善，但CLI工具和MCP集成需要自己封装，坑比较多。

### 踩坑记录

- **Git Co-author问题**: Cursor会自动添加`Co-authored-by`，需要用`git filter-branch`清理历史
- **Task构造函数**: v7更新后Task需要task_id参数，老版本测试会fail
- **异步状态**: PilotRouter中异步编排会导致state快速跳转，测试断言需要考虑多种可能状态

### 好用的写法

- **工厂模式**: `default_orchestrator_service()`统一依赖注入
- **Builder模式**: ContextPack.builder()链式构建复杂对象
- **策略模式**: ReasoningPattern根据context自动选择推理方式

### AI使用环节

- **代码生成**: 70%的boilerplate代码由AI生成，效率很高
- **测试用例**: AI很善于生成edge case和异常场景测试
- **文档写作**: README和架构文档的结构化内容AI辅助很好
- **重构**: 大规模rename和结构调整AI处理得很稳定

### Prompt调优技巧

- **分层描述**: 先说整体架构，再说具体实现，最后说边界处理
- **举例说明**: 用已有代码举例，让AI理解代码风格和模式
- **约束明确**: 明确说明不要改哪些文件，遵循哪些原则

### 折腾出来的小工具

- **快速测试**: `PYTHONPATH=. python -m pytest tests/test_pilot_router.py -v`
- **状态检查**: `git log --oneline -10` 快速查看提交历史  
- **API测试**: `curl localhost:8001/v7/api/pilot/status` 检查服务状态

**总结**: 这个周期最大的收获是真正实现了一个end-to-end的AI系统，不是demo而是可用的产品。从技术栈选择到架构设计，从开发流程到部署运维，都有了完整的实战经验。AI协作让开发效率提升很多，但人的架构思维和业务理解还是不可替代的。
