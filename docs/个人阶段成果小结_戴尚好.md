### 三、过程复盘

**1. 环节与方法**

- 读透赛题文档（`基于IM的办公协同智能助手_赛题公开版整理.md`）+ v1 代码 + 飞书开赛仪式纪要，列出 Must-have vs Good-to-have 全量差距表。
- 先**架构澄清**（Plan 模式 3 个问题：客户端栈/生成引擎/优先级），避免盲干。
- 按 **5 周期 × 3 天** 节奏切任务，每周期都有可独立演示的产出。
- 采用**双栈 fallback 策略**：LLM Planner + Heuristic Planner、y-py CRDT + plain JSON broadcast、飞书 Docx API + 本地 markdown，每一条都保证离线也能 demo。

**2. 不顺与破局**

- Planner 接了 Advanced 澄清功能后，老用例 `test_planner_accepts_llm_output` 顺序断言挂了。破局方案：暴露 `allow_clarify=False` 开关给调用方显式控制，既保留了加分项又不破旧 API。
- `core/sync/crdt_hub.py` 的 `y_py` 是可选依赖；在没有装的机器上需要优雅降级，最终抽象成"CRDT 分支 + 透明 relay 分支"两条路径。

**3. 可复用沉淀**

- **场景注册模式**：`ScenarioRegistry` 让评委可以通过 `/api/pilot/scenarios` 看到 A-F 六个场景全覆盖，后续扩展只需注册一个 `Scenario` dataclass。
- **工具 registry 模式**：所有 `tools/*.py` 遵守 `(step, ctx) -> dict` 契约；测试里只要 monkey-patch registry 即可替换为假工具。
- **Orchestrator 的事件总线**：把 `ExecutionEvent` 作为唯一真相，Bot、Dashboard、Flutter 都订阅同一流，没有第二套状态机。

### 四、本周随手记

- 飞书 Docx API 的 block children 接口，heading1/heading2/heading3 分别对应 block_type 3/4/5，bullet 是 12；踩过一个坑：必须构造 `Text(elements=[TextElement(text_run=TextRun(...))])` 才能渲染，否则 block 变空白。
- Slidev markdown 是有 `---` 分隔符的，解析时要注意 YAML front-matter 会和 slide 分隔符混淆。
- Flutter 四端一套代码最重要的是 `NavigationRail`（桌面端）+ `NavigationBar`（移动端）在 `AppShell` 里通过 `MediaQuery.size.width` 切换，不需要写两套 UI。

### 三、关键反思

- **「多 agent 是质量放大器」是本周期最大的认知升级**：Anthropic 内部数据显示 Lead + Sonnet Workers 比单独 Opus 提升 **90.2%**。arxiv:2508.17536 进一步揭示：增益的主要来源是 **多数投票**（majority voting），不是单纯 debate 本身。所以我们的设计是：3 模型并行生成 + judge 汇总 + 不收敛才 debate。
- **Builder-Validator 分离是避免盲点的关键**：让写代码的人不能审自己的代码。v4 每个关键输出都有**独立** Validator subagent（Critic / Citation / Risk / Security / A11y），永不重叠。
- **「harness engineering」胜过「model engineering」**：Nous 的 Hermes 哲学：**LLM 可替换，harness 才是工程本体**。我们把 4 个模型当成可热插拔组件，规划用 MiniMax M2.7，中文用豆包，审查用 DeepSeek 省钱，长文档用 Kimi。
- **不做向量 DB 是正确选择**：Hermes 用 SQLite + FTS5 达到 10ms 级召回，fully inspectable 且易 debug。pgvector/Pinecone 对我们这种办公场景是过度工程。
- **Named Agents 比 prompt 切换优雅**：每个 agent 一个 yaml 文件，独立 model/tools/memory_scope/instruction。用户无感自动路由，显式则 `@pilot 做方案`。这是 ShanClaw 带给我的最佳实践。

### 三、关键反思

- **"代码可以重构"是最关键的用户指令**：之前一直在打补丁，直到用户明说允许重构后才放开手脚，一次性把 orchestrator 换成 LangGraph、把 Flutter 画布改成 WebView 真协同。
- **对标 Claude Code 的最大收获不是某一个模块，而是 harness 心态**：Context / Permission / Hook / Skill / Subagent 不是可选项，而是"让 agent 不抽风"的必要基建。
- **飞书 MCP 远程 + 本地双通道是被低估的**：远程 MCP 把云文档原生能力白送给我们，本地 MCP 把 2500+ API 零写代码接入，这两者都比我们自己包 SDK 快一个数量级。
