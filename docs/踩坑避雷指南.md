# 飞书AI Agent开发踩坑避雷指南

> 基于 LarkMentor v4 重构过程中的实战踩坑经验总结

## 🔥 重大架构踩坑

### 1. y-websocket协议实现陷阱

**坑点**：自以为懂WebSocket，自制实时协作协议  
**症状**：客户端连接正常，但CRDT同步完全不工作  
**根因**：y-websocket有标准的二进制消息格式，不是普通JSON  

```python
# ❌ 错误：自制协议
async def websocket_handler(websocket):
    await websocket.send(json.dumps({"type": "sync", "data": doc_state}))

# ✅ 正确：标准y-websocket
from y_py import YDoc
from ypy_websocket import WebsocketProvider

def setup_yws_server():
    return WebsocketProvider(YDoc(), websocket)
```

**避雷建议**：直接用 `ypy-websocket` 库，不要自己实现

---

### 2. 飞书MCP配置地狱

**坑点**：官方MCP文档不全，stdio vs HTTP混用  
**症状**：MCP工具注册成功，但调用时各种超时/格式错误  
**根因**：本地`@larksuiteoapi/lark-mcp`用stdio，远程`mcp.feishu.cn`用HTTP

```json
// ❌ 错误：混用协议
{
  "local_mcp": {
    "transport": "http",  // 应该是stdio
    "command": ["npx", "@larksuiteoapi/lark-mcp"]
  }
}

// ✅ 正确：分离配置  
{
  "stdio_servers": {
    "lark_local": {
      "command": ["npx", "@larksuiteoapi/lark-mcp"],
      "presets": ["default", "im", "doc"]
    }
  },
  "http_servers": {
    "lark_remote": {
      "url": "https://mcp.feishu.cn/mcp",
      "headers": {"Authorization": "Bearer xxx"}
    }
  }
}
```

**避雷建议**：本地用stdio，远程用HTTP，不要混

---

### 3. 飞书Card 2.0 element_id陷阱  

**坑点**：Card流式更新总是失败，element_id找不到  
**症状**：卡片创建成功，但PATCH更新返回400错误  
**根因**：element_id必须在初次卡片JSON中预定义

```python
# ❌ 错误：后添加element_id
def create_card():
    card = {"type": "template", "data": {"content": "思考中..."}}
    send_card(card)
    # 稍后尝试更新，element_id不存在
    update_card_element("thinking_text", "新内容")  # 失败

# ✅ 正确：预定义所有element_id
def create_thinking_card():
    card = {
        "type": "template", 
        "data": {
            "content": {
                "type": "div",
                "element_id": "thinking_text",  # 预定义
                "text": {"type": "plain_text", "content": "思考中..."}
            }
        }
    }
    return card
```

**避雷建议**：所有要动态更新的元素，初始化时就定义element_id

## 🛠️ 技术实现踩坑

### 4. SQLite FTS5中文分词失效

**坑点**：英文搜索正常，中文搜索结果为空  
**症状**：`SELECT * FROM memory WHERE content MATCH '智能助手'` 返回0行  
**根因**：FTS5默认按空格分词，中文需要额外分词器

```python
# ❌ 错误：直接用FTS5
CREATE VIRTUAL TABLE memory USING fts5(content, title);

# ✅ 正确：集成jieba分词
import jieba

def setup_fts5_chinese():
    # 方案1：预分词存储
    content_segmented = " ".join(jieba.cut(content))
    cursor.execute("INSERT INTO memory VALUES (?, ?)", (content_segmented, title))
    
    # 方案2：查询时分词
    query_segmented = " ".join(jieba.cut(query))
    cursor.execute("SELECT * FROM memory WHERE content MATCH ?", (query_segmented,))
```

**避雷建议**：中文FTS一定要分词，推荐预处理方案

### 5. FastAPI multipart依赖缺失

**坑点**：语音上传接口突然500错误  
**症状**：`RuntimeError: Form data requires "python-multipart" to be installed`  
**根因**：FastAPI的`UploadFile`需要额外依赖

```python
# ❌ 缺失依赖
from fastapi import UploadFile, File, Form
# 运行时报错

# ✅ 正确安装
pip install python-multipart>=0.0.9

# requirements.txt添加
python-multipart>=0.0.9
```

**避雷建议**：用到文件上传就装multipart，不要等报错

### 6. Docker多架构Node.js版本冲突

**坑点**：Docker build在M1 Mac正常，部署到x86服务器失败  
**症状**：`@larksuite/cli` 安装报错，native依赖编译失效  
**根因**：Node.js版本与CPU架构不匹配

```dockerfile
# ❌ 错误：固定版本
FROM node:18-alpine

# ✅ 正确：多架构适配
FROM --platform=linux/amd64 node:18-alpine
RUN npm config set target_arch x64
RUN npm config set cache /tmp/.npm
```

**避雷建议**：Docker指定平台，避免架构不一致

## 🚨 Agent协作踩坑

### 7. 子Agent无限递归

**坑点**：Agent调用子Agent，子Agent又调用子Agent...  
**症状**：内存爆炸，请求超时  
**根因**：没有递归深度限制和上下文隔离

```python
# ❌ 错误：无限递归
def run_agent(task, context):
    if need_help:
        sub_result = run_agent(subtask, context)  # 递归调用

# ✅ 正确：深度限制+隔离
def run_agent(task, context, max_depth=1):
    if context.depth >= max_depth:
        return "达到最大深度，终止"
    
    isolated_context = context.copy()
    isolated_context.depth += 1
    return run_subagent(subtask, isolated_context)
```

**避雷建议**：子Agent必须限制深度，隔离上下文

---

### 8. 多模型并发调用成本失控

**坑点**：为了提升质量，同时调用4个模型，成本爆炸  
**症状**：一次对话消耗几十元，预算快速耗尽  
**根因**：没有成本预算和智能路由

```python
# ❌ 错误：无脑并发
async def multi_model_call(prompt):
    tasks = [
        call_doubao(prompt),
        call_minimax(prompt), 
        call_deepseek(prompt),
        call_kimi(prompt)
    ]
    results = await asyncio.gather(*tasks)  # 成本×4

# ✅ 正确：预算路由
def smart_route(task_type, budget_left):
    if budget_left < 1.0:  # 低成本
        return "deepseek"
    elif task_type == "creative":
        return "minimax"  
    else:
        return "doubao"
```

**避雷建议**：多模型要有成本控制，按场景路由

## 🔐 安全权限踩坑

### 9. Tool权限检查绕过

**坑点**：以为加了权限检查就安全，实际可被绕过  
**症状**：Agent执行了危险操作，如删除文件  
**根因**：权限检查在错误的层级，可通过子Agent绕过

```python
# ❌ 错误：只在入口检查
@tool
def delete_file(path):
    # 没有权限检查
    os.remove(path)

def agent_loop(user_input):
    if not check_permission(user_input):  # 只检查输入
        return "无权限"
    # 但Agent可以自己调用delete_file

# ✅ 正确：工具级检查
@tool  
def delete_file(path, user_id):
    if not has_permission(user_id, "file.delete"):
        raise PermissionError("无删除权限")
    os.remove(path)
```

**避雷建议**：权限检查要在工具执行时，不是入口时

### 10. Secret泄露到日志

**坑点**：调试时打印所有参数，API密钥被记录  
**症状**：日志文件包含敏感信息  
**根因**：没有敏感信息过滤

```python
# ❌ 错误：裸日志
def call_api(api_key, data):
    logger.info(f"调用API: {api_key}, {data}")  # 泄露密钥

# ✅ 正确：脱敏日志
def call_api(api_key, data):
    safe_key = api_key[:8] + "***" if api_key else "None"
    logger.info(f"调用API: {safe_key}, {data}")
```

**避雷建议**：日志脱敏，永远不要打印完整密钥

## 🎯 比赛特定踩坑

### 11. Demo录制音频不同步

**坑点**：屏幕录制正常，但声音延迟或缺失  
**症状**：Demo视频没声音或声画不同步  
**根因**：Mac系统权限限制麦克风访问

```bash
# ✅ 解决方案
# 1. 系统偏好设置 → 安全性与隐私 → 隐私 → 麦克风
# 2. 允许录制软件访问麦克风
# 3. 使用专业录制软件如 ScreenFlow
```

**避雷建议**：Demo前先测试录制，检查音频权限

### 12. 技术报告LaTeX编译失败

**坑点**：中文LaTeX模板编译各种错误  
**症状**：`xelatex`编译中断，字体找不到  
**根因**：中文字体路径和编码配置错误

```latex
% ❌ 错误：硬编码字体路径
\setCJKmainfont{/System/Library/Fonts/PingFang.ttc}

% ✅ 正确：系统字体名
\setCJKmainfont{PingFang SC}
\setCJKsansfont{PingFang SC}  
\setCJKmonofont{Menlo}
```

**避雷建议**：用字体名，不要用绝对路径

## 💡 最佳实践总结

### 开发阶段
1. **先跑通最小可用版本**，再追求功能完整
2. **每个模块都有单元测试**，集成前先单测
3. **日志分级详细**，便于排查问题
4. **配置外部化**，不要硬编码

### 集成阶段  
1. **API先mock再实现**，避免外部依赖阻塞
2. **异常处理要全面**，每个外部调用都要try-catch
3. **超时时间要合理**，防止无限等待
4. **重试机制要指数退避**，避免雪崩

### 部署阶段
1. **蓝绿部署必备**，回滚要快
2. **健康检查端点**，监控服务状态  
3. **资源限制配置**，防止OOM
4. **日志轮转**，防止磁盘满

### 比赛阶段
1. **Demo脚本要反复练习**，确保流畅
2. **备用方案准备**，主功能失效时的降级
3. **视频录制多备份**，防止文件损坏
4. **技术报告要有量化数据**，不只是功能列举
