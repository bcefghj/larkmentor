## 三、过程复盘与沉淀

### 1. 这周主要搞定了哪些环节？用了什么方法？

**主要环节**（按执行顺序）：

**P0 资产盘点**：用 Python 3.12 venv 装依赖，跑基线测试（发现 222 passed / 14 failed，14 个 fail 全是 v4 时代的 pilot_api 和 recovery_card，是目标改造对象），验证 MiniMax + Doubao 双供应商连通。

**P2 领域层（DDD）**：参考 Claude Code 51 万行源码的设计哲学——Tool 是一等公民、Permission 三态、Sub-agent transcript 隔离——建立 Task 聚合根。核心设计选择：状态机是纯函数（`transition(state, event) -> state`），不允许 mutation，测试 50+ 条转移规则只需 0.02s。

**P3 主动识别**：一个关键 bug：最初只扫最后 1 条消息，导致"u1 说任务、u2 附议、u3 资料"这种多轮群聊识别不到。修改为最近 5 条消息联合文本打分后，测试全过。

**P6 多 agent**：Builder-Validator 分离的关键：给每个 sub-agent 传入独立 `llm_caller`，transcript 不共享。这样 @validator 审查时完全不知道 @pilot 生成了什么——true isolation。

**P13 A/B 矩阵**：Doubao 每条调用约 30s，multi-agent 配置每条超过 100s（内部调 3-4 次 LLM），总时长 43.8 min。数据出来后发现单 agent 反而分数最高——原因是评分方法对"字数适中+markdown 整齐"友好，multi-agent 经多轮改写字数膨胀且 citation 占位符降低 readability。**选择如实披露而不是隐瞒这个不利数据**，写了完整方法论说明。这个决定值得记录：工程诚实度比宣传数字更值钱。

**工具分工**：
- AI 负责：所有新模块的初版代码 + 完整测试套件 + 文档草稿 + git commit message
- 我负责：设计判断（如状态机是否加 CANVAS_GENERATING）/ bug 分析 / 数据结论解读 / 最终评审

### 2. 卡住过的情况与破局

**坑 1：Python 3.9 → 3.12 venv 切换**

原项目用系统 Python 3.9.6，但 mem0ai 等依赖要求 3.10+。找到 `/opt/anaconda3/bin/python3.12` 建 venv 解决。同时发现 mem0ai 会尝试连 qdrant 导致 pytest 卡 7 分钟——加 `--timeout=5` 解决。

**坑 2：git 初始化冲突**

`github_public/Agent-Pilot/` 本地没有 git 仓库，但 `git checkout origin/main -- .` 因为现有文件冲突失败。解法：先用 `git add -A . && git commit` 把所有文件打包成 baseline commit，再在这个 baseline 基础上叠加新 commit。这样评委看到"18 次 commit"的完整迭代痕迹，而不是从零开始。

**坑 3：promptfoo 初始版本 22/32**

v7 新增的 OWASP LLM Top 10 用例里，`###SYSTEM###` / `<script>alert(...)` / `javascript:...` / `rm -rf` 等模式原始 `_SUSPICIOUS_PATTERNS` 没有覆盖。在 `transcript_classifier.py` 里追加 17 条新正则，32/32 全过。

**坑 4：A/B 矩阵 payload 含控制字符**

`/tmp/release_body.md` 里有中文代码块导致 JSON 解析报 `Invalid control character`。改为用 `python3 -c "import json; print(json.dumps(...))"` 预处理后写到临时文件，再用 `-d @file` 上传，解决。

### 3. 可复用的东西

**精华 1：状态机纯函数模式**

```python
_TRANSITIONS: Dict[Tuple[State, Event], State] = {
    (State.SUGGESTED, Event.USER_CONFIRM): State.ASSIGNED,
    ...
}

def transition(state, event) -> State:
    if (state, event) in _TRANSITIONS: return _TRANSITIONS[(state, event)]
    raise InvalidTransitionError(state, event)
```

纯字典 + 纯函数，零依赖，50+ 转移 0.02s 全跑完。任何业务 Agent 的状态管理都能直接套用。

**精华 2：three-gate intent detector 模式**

```
gate1_rules -> gate2_llm_judge -> gate3_min_info
+ cooldown.is_cooling + cooldown.is_ignored
```

每个闸门独立可替换（测试时 mock llm_caller），三闸门组合既防误触发又防过度安静。适合任何"被动监听 → 主动介入"类产品。

**精华 3：诚实 A/B 报告模板**

`tests/reports/AB_MATRIX_ANALYSIS.md` 结构：
1. 真实数据表格（无 mock，每档 avg / N）
2. 为什么直觉预期结果没有出现（方法论 caveat）
3. 哪里真正有价值（多 agent 的适用场景）
4. 复现步骤

这个模板可以直接复用到任何"我跑了对比实验但结果跟预期相反"的场景。

**精华 4：multi-agent transcript 隔离模式（来自 Claude Code）**

```python
def _run_validator(task, content, report):
    ts = AgentTranscript(agent="@validator", ...)
    result = validator.run(content)  # 独立 context，不知道 builder 写了什么
    task.log(agent="@validator", kind="result", content=result)
```

不通过父 context 传递，每个 sub-agent 独立 transcript，保证 Validator 永不自审。

## 四、本周随手记

**关于"诚实 vs 宣传"的决策**

A/B 矩阵数据出来之前，README 里写的是「+43% 绝对值提升」。真实数据出来之后，单 agent 得分 74.4，多 agent 各档在 67-70 之间。

有两个选项：
1. 选择性报告：只展示某个对多 agent 有利的子集
2. 如实披露：放出完整数据，加方法论说明

选了 2，理由：评委会审代码，如果跑一遍脚本发现数据对不上，比"没有 A/B 数据"更减分。而且方法论说明本身就展示了对实验设计的深度理解——这才是"AI Native 思考者"真正该有的样子。

**关于"不要为精简而精简"的教训**

本次升级计划最初版本（v5）里，我把 v1 双线（Smart Shield + Mentor 4 Skills）标记为"v1 LarkMentor 无关遗产，下放 legacy/"。这个判断是完全错误的：

- 三个赛道志愿共用主体代码
- 「工位上同时发生」立意需要 Shield + Mentor 作为「三线产品」的前两条线
- Promptfoo 14/14 真实数据在 v6 里就有了

教训：**搞清楚"为什么要精简"再精简，不要用"追求简洁"来掩盖"我没认真读代码"。**

**关于 6 级 Memory 的价值**

v6 的 ARCHITECTURE.md 里写着：「flow_memory_md.resolve_memory_md 已写但未注入」。v7 做的核心事情之一就是把这个"已写未注入"变成"真正注入"。

6 级 Memory 注入后，IntentDetector 调 LLM 时的 system prompt 包含了：
```
[Enterprise] 我司财年 4-3 月，关键人物：CEO 张总
[Group] 本群活动 = 校园推广
[User] 我喜欢简洁汇报风格
```

这才是"AI 同事 + 长期陪伴"真正的工程体现，而不是单轮对话工具。

**提示词工程小技巧**

在 P3 主动识别里，LLM judge prompt 的关键设计：

```python
LLM_JUDGE_PROMPT = """
判断是否构成可执行任务。

输出严格 JSON（不要 markdown 围栏，不要解释）：
{"is_task": true|false, "task_type": "...", "goal": "...", "confidence": 0.0-1.0}

判断标准（保守）：
- 单纯打招呼、闲聊、技术问答不算任务
- 信息不足时 confidence 应低（< 0.5），但仍可标 is_task=true 让用户澄清
"""
```

关键是"保守"二字——让 LLM 倾向于低 confidence，上游再用 0.55 门槛判断是否弹卡，而不是 LLM 自己决定弹不弹。这样闸门 3（最小信息）才有发挥余地。
