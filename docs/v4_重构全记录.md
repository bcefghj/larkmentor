## 重构前痛点分析

### 1. 架构层面
- **分裂设计**：Smart Shield、Mentor、Agent-Pilot 三个独立引擎，缺乏统一性
- **伪工具化**：大量模拟工具调用，实际无法执行
- **内存割裂**：无跨会话记忆，每次对话重新开始

### 2. 工程层面
- **在线依赖**：Flutter客户端依赖CDN，离线功能假假的
- **协议错误**：WebSocket实现不是标准y-websocket协议
- **安全缺失**：无权限控制、无速率限制、无审计日志

### 3. 比赛层面
- **覆盖不全**：6个必选场景实现不完整
- **加分项空**：4个加分项基本没做
- **Demo弱**：缺乏评委wow点，难以展示Agent能力

## v4重构方案：统一Agent Harness

### 核心架构决策

1. **Claude Code式Agent Loop**：9步pipeline，统一所有AI交互
2. **工具真实化**：70+真实工具，替代所有模拟调用  
3. **飞书官方栈**：MCP + CLI Skills + Card 2.0，深度集成
4. **多Agent协同**：Orchestrator-Worker + 5种推理模式
5. **企业级安全**：7层权限控制 + 审计 + 限流

## 踩坑记录

### 技术坑
1. **y-websocket协议理解偏差**：最初自制WebSocket，后改用标准库
2. **MCP配置复杂性**：stdio vs HTTP双通道，预设参数调试
3. **Card 2.0 element_id机制**：流式更新需要精确定位DOM元素
4. **SQLite FTS5中文分词**：需要额外配置jieba分词器

### 架构坑  
1. **工具注册顺序依赖**：部分工具需要特定初始化顺序
2. **子Agent上下文隔离**：避免内存污染和无限递归
3. **权限检查性能**：7层检查需要缓存优化
4. **多模型调用成本**：需要精确的token计算和预算控制

### 部署坑
1. **Docker层依赖**：Node.js + Python + CLI工具版本冲突
2. **蓝绿切换时机**：数据库迁移与服务切换的原子性
3. **飞书Webhook重试**：需要幂等性处理重复事件
4. **Redis连接池**：多Agent并发访问的锁竞争
