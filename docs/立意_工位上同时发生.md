## 1 · 核心命题（300 字标准版）

> 在飞书这样的 IM 协同环境里，每个人每天都在和两件事较劲——
>
> 一件是「**这条消息现在该不该让我看到**」。重要 @ 和噪音 @ 用同样的方式震动你的手机，你被迫每条都点开判断。
>
> 一件是「**这条消息我该用什么方式发出去**」。新人在群里输入框前面犹豫 30 秒删了 3 遍——"好的"够不够正式？"收到"会不会显得敷衍？
>
> 这两件事**发生在同一个工位上、同一个人身上、同一段时间内**。但今天打开任何一款飞书插件市场，做静音的不管表达，做写作的不管打扰——没有产品意识到这两件事本来就该一起解。
>
> **LarkMentor 是把这两件事放在同一个 IM Bot 里的一次工程尝试**——一边守消息层，一边守表达层，背后是同一份「你这家公司怎么做事」的知识。

## 2 · 200 字介绍 5 个备选版本

### v1 · 质问开场版（最锋利）

> 在飞书里，每个人每天都在和两件事较劲。
>
> 第一件——**这条消息现在该不该让我看到？** 重要 @ 和噪音 @ 用同样的方式震动你的手机，你不点开就不知道是谁在叫你。
>
> 第二件——**这条消息我该用什么方式发出去？** 新人在群里输入框前面犹豫半分钟、删了 3 遍——「好的」够不够正式？「收到」会不会显得敷衍？
>
> 这两件事**发生在同一个工位上、同一个人身上、同一段时间内**。但今天打开飞书插件市场，做静音的不管表达，做写作的不管打扰——**没有人意识到这两件事本来就该一起解。**
>
> LarkMentor 把它们放在同一个 IM Bot 里——一边守消息层，一边守表达层，背后是同一份「这家公司怎么做事」的知识。

- **调性**：质问 → 留白 → 答案
- **风险**：「没有人意识到」需要 hold 住"凭什么是你意识到"的追问
- **适合场景**：答辩开场 / Pitch 首页

### v2 · 数据排比开场版（最稳）

> 知识工作者每被打断一次，平均需要 **23 分 15 秒** 才能回到原来的工作状态（UC Irvine, 2008）。
> 飞书企业用户日均收发消息 **数百条**，其中**约 80%** 不需要立刻回复。
> 新员工平均需要 **8-12 个月** 才能达到全产能（SHRM, 2023）。
> 而企业里 **80%+** 的关键知识是隐性的——存在老员工脑子里，从未被写下来（Nonaka, 1995）。
>
> 这 4 个数字说的是同一件事：**飞书 IM 上每个人每天都在被两件事消耗——不该看的消息和不会说的话**。
>
> 但今天没有产品把这两件事一起解。LarkMentor 把它们放在同一个 IM Bot 里——一边守消息层，一边守表达层，背后是同一份「这家公司怎么做事」的知识。

- **调性**：数据 → 收束 → 答案
- **风险**：开头 4 行数据可能学术化过头
- **适合场景**：⭐ README 第一段 / 报名表"具体问题" / PDF 封面页

### v3 · 行业现状对比版（最商业感）

> 飞书生态里，「免打扰」类工具有几十款，「AI 写作」类工具也有几十款。
>
> 但你打开任何一款，都是单点解：免打扰只管挡，写作只管写。**没有产品意识到——一个人在工位上被打扰的那一刻，往往也是他需要替自己说点什么的那一刻**。
>
> 重要 @ 来了你要回，临时拉会你要应，新项目@ 你要确认任务——**每一次「该不该被打断」的判断，都连着一次「该怎么说出去」的选择**。
>
> LarkMentor 把这两件事做在同一个 IM Bot 里——Smart Shield 守消息层，Mentor 守表达层，背后是同一份「这家公司怎么做事」的知识。
>
> 不是又一个免打扰，也不是又一个 AI 写作。**是把它们本来就该合在一起的，第一次合在一起。**

- **调性**：竞品扫描 → 缺口指认 → 答案
- **风险**：依赖"飞书插件市场没人合做"这个事实——需要竞品扫描表佐证
- **适合场景**：评委版 PDF 第二页（紧跟封面页之后的"市场缺口"小节）

### v4 · 研究生口吻版（最贴身份）

> 我做这个项目是因为观察到一件事——**飞书 IM 上每个人每天都在和两件事较劲**：
>
> 一件是 **「这条消息现在该不该让我看到」**——重要的和噪音用同样的方式震动手机；
>
> 一件是 **「这条消息我该用什么方式发出去」**——新人输入框前犹豫半分钟、删 3 遍。
>
> 我去翻飞书插件市场，发现做免打扰的几十家、做 AI 写作的几十家，**但没人把这两件事做在一起**。
>
> 我觉得这两件事**本来就发生在同一个工位上、同一个人身上、同一段时间内**——应该一起解。
>
> LarkMentor 是我对这个想法的工程实现：一个飞书 IM Bot，一边守消息层，一边守表达层，背后是同一份「这家公司怎么做事」的知识。**目前是 demo + 自测阶段，重点交付架构和工程能力。**

- **调性**：第一人称观察 → 调研发现 → 想法 → 实现
- **风险**：极低，研究生说"我观察到、我做了一个 demo"非常 honest
- **适合场景**：⭐ 报名表"AI 项目/作品/研究" / 个人版 PDF / 简历项目描述

### v5 · 短而锋利版（最适合一句话钩子）

> 飞书 IM 上，每个人每天都在和两件事较劲——**「这消息现在该不该看」** 和 **「我该用什么方式说出去」**。
>
> 静音工具只管前者，AI 写作只管后者。但**这两件事发生在同一个工位上**——重要 @ 进来的那一刻，往往也是你要替自己说点什么的那一刻。

- **调性**：短促有力
- **风险**：信息密度低
- **适合场景**：60 秒电梯演讲 / Twitter 介绍 / 名片背面

## 3 · 双线"合体机制"3 个真实交点

立意「工位上同时发生」最大的弱点是"听起来还是两个功能"。必须给一个**让两条线在同一个机制上真合体**的解释——而且必须是**从已有代码里长出来的**。

### 合体点 1 · 同一份"组织默契知识"被两边共用

**机制**：守门员判断"这消息该不该打断你"和起草员判断"这话该怎么说"，**用的是同一份用户级 RAG**。

**对应代码**：
- `core/mentor/knowledge_base.py` 用户级 RAG（已有）
- `core/classification_engine.py` 6 维分类时也调用这份知识做"组织语境"判断（**计划升级**）
- `core/mentor/mentor_write.py` 写作时也调用这份知识做"组织风格"对齐（已有）

**讲法**：
> "**两条线下面是同一份知识——`knowledge_base.py` 既被 Smart Shield 调（判断这家公司什么算紧急），也被 Mentor 调（判断这家公司怎么说话）**。一份知识，两条服务线。"

### 合体点 2 · Recovery Card（双线在 UI 上唯一交点）⭐ 最直观

**机制**：你被打断后回到工位，Smart Shield 给你弹一张"恢复卡片"——
- 上半张是"刚才挡掉的消息摘要"（专注力线产物）
- 下半张是"AI 替你起草的回复草稿"（小白线产物）
- 一张卡，双线合体

**对应代码**：
- 现有：`core/context_recall.py` 上下文召回基础
- 现有：`core/mentor/proactive_hook.py` 起草草稿
- **计划新增**：合并这两个产物到一张卡片

**讲法**：
> "**Recovery Card 是双线唯一的 UI 交点**——上半张'我替你挡了什么'，下半张'我替你起草了什么'。一张卡解决'回到上下文 + 怎么回话'两件事。"

### 合体点 3 · 同一份 FlowMemory 在持续学习

**机制**：每次 Smart Shield 判断对/错（用户改了白名单/紧急词），每次 Mentor 起草被采纳/拒绝（用户改了草稿），**这些信号都喂回同一份 FlowMemory**。两条线**在持续学习同一份默契**。

**对应代码**：
- `core/sender_profile.py` Shield 学发件人
- `core/flow_memory/working.py` 三层记忆
- `core/mentor/growth_doc.py` Mentor 写入成长档案

**讲法**：
> "**两条线越用越懂同一家公司**——Shield 学'谁是真紧急'，Mentor 学'怎么说才得体'，学到的东西沉淀到同一份 FlowMemory。**用得越久，AI 越像这家公司的老员工。**"

| 志愿 | 切片用法 | 加的材料 |
|---|---|---|
| **1 · IM 协同助手** ⭐ | 主体 = 这份立意 + 双线合体机制 | README + 产品介绍 + 答辩稿 |
